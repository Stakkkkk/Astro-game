# Заметка к кейсу: публичный веб

Дата: `2026-07-04 13:33:17 +07:00`.

## Ситуация

Пользователь сказал «делаем по плану» после обсуждения выхода из локального запуска в реальный веб. Это момент, когда план уже достаточно согласован, поэтому агент должен не пересказывать его заново, а довести до публичной ссылки, проверок и понятного остаточного ручного шага.

## Методические наблюдения

- Публичная ссылка меняет восприятие проекта: прототип перестает быть «локальной штукой на машине разработчика» и становится демонстрируемым продуктом.
- Для человека без ИТ-бэкграунда важнее короткая проверяемая формула «вот ссылка, вот как понять что она живая», чем подробность про все слои инфраструктуры.
- Инфраструктурные блокеры нужно превращать в конкретные ручные шаги. В этом случае постоянный Pages deploy потребовал Cloudflare auth, но официальный `wrangler deploy --temporary` позволил получить рабочий preview.
- Если один официальный путь требует больше подготовки, стоит проверить другой официальный путь. Workers Static Assets позволил отдать Vite-клиент и Durable Object WebSocket с одного домена.
- WebSocket-проект нельзя считать опубликованным только по HTML-странице. Нужен smoke против публичного `wss://` endpoint.

## Практические приемы

- Для публичного MVP сначала получить временный preview, потом оформлять постоянный deploy.
- Фиксировать URL, health endpoint, WebSocket endpoint и команды проверки в README.
- Оставлять локальный Node adapter, если он помогает быстро тестировать игровой loop без Cloudflare runtime.
- Выносить game core до деплоя, чтобы Node adapter и Worker adapter не начали расходиться в правилах игры.
- Документировать оставшийся ручной шаг без драматизации: `wrangler login` или `CLOUDFLARE_API_TOKEN` для постоянного деплоя.
