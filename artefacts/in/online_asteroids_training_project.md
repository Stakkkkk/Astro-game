# Учебный проект: сетевая игра в духе Asteroids

## 1. Краткая идея

Нужно разработать учебный проект: **онлайн-игру по мотивам классической Asteroids-механики**.

Игрок управляет маленьким космическим кораблём-стрелочкой, летает в условно бесконечной или очень большой 2D-локации, стреляет по астероидам, разбивает их, получает очки, бонусы и штрафы. Игра изначально проектируется как **сетевая**, не как локальная одиночная демка.

Первый режим — **кооперативный PvE**: несколько игроков находятся в одной сессии, совместно уничтожают астероиды, набирают очки и соревнуются внутри текущей игровой сессии.

## 2. Главная цель проекта

Сделать не коммерческую игру, а **учебный проект**, на котором можно пройти полный цикл разработки:

- игровой цикл;
- физика и коллизии;
- клиентская отрисовка;
- сетевое взаимодействие;
- авторитетный сервер;
- синхронизация состояния между игроками;
- деплой в интернет;
- минимальный DevOps;
- постепенное развитие проекта от MVP к более интересной игре.

## 3. Ключевое требование

Игра должна быть **онлайн-сетевой**.

Это означает:

- не просто локальная игра в браузере;
- не только hot-seat;
- не только LAN;
- не мок сетевого режима;
- не одиночная игра с будущей «заглушкой» под сеть.

Уже в MVP несколько игроков должны иметь возможность подключиться к одной игровой комнате через интернет и видеть общее игровое состояние.

## 4. Предпочтительная платформа

### 4.1. Первый приоритет — веб-версия

Предпочтительный вариант реализации — **браузерная игра**.

Причины:

- минимальный порог входа;
- не нужно устанавливать мобильное приложение;
- проще тестировать;
- проще дать ссылку другому человеку;
- удобнее использовать как учебный проект;
- проще деплоить через GitHub + Cloudflare-стек или аналогичный serverless/edge-подход.

Ориентировочная схема:

```text
Browser Client
  ↓ WebSocket
Online Game Server / Room Coordinator
  ↓
Session State / Room State
```

Кандидатная технологическая линия:

```text
Frontend:
  TypeScript
  Canvas 2D или PixiJS / Phaser

Backend:
  TypeScript
  WebSocket
  Cloudflare Workers + Durable Objects
  или Node.js WebSocket server

Deploy:
  GitHub repository
  Cloudflare Pages для клиента
  Cloudflare Workers / Durable Objects для игрового сервера
```

### 4.2. Второй приоритет — мобильное приложение

Мобильное приложение рассматривается только если веб-вариант окажется нецелесообразным.

Если агент предлагает мобильную реализацию, он обязан объяснить:

- почему веб-версия хуже;
- как будет реализован онлайн;
- где будет жить сервер;
- как тестировать приложение без лишней боли;
- как минимизировать стоимость и сложность разработки.

В мобильном варианте предпочтительно рассмотреть кроссплатформенный стек:

- Flutter;
- React Native;
- Godot export;
- Unity WebGL/mobile;
- другой вариант, если агент докажет целесообразность.

## 5. Игровой цикл MVP

Минимальная игровая сессия:

1. Игрок открывает ссылку на игру.
2. Вводит ник.
3. Создаёт комнату или подключается к существующей.
4. Появляется на карте в виде маленького корабля.
5. Управляет кораблём:
   - поворот влево/вправо;
   - ускорение;
   - торможение или инерционное движение;
   - стрельба.
6. В общей локации появляются астероиды.
7. Игроки стреляют по астероидам.
8. Астероиды получают урон, разрушаются или делятся на меньшие части.
9. Игроки получают очки.
10. При столкновении с астероидом игрок получает урон.
11. При смерти игрок теряет очки или получает штраф.
12. В рамках сессии ведётся рейтинг.
13. Сессия может быть перезапущена.

## 6. Базовые игровые сущности

### 6.1. Player / Ship

Поля:

- id;
- nickname;
- position: x, y;
- velocity: vx, vy;
- rotation;
- health;
- score;
- alive/dead;
- lastInputSequence;
- connectionState.

Поведение:

- движение с инерцией;
- поворот;
- ускорение;
- стрельба;
- получение урона;
- смерть;
- респавн.

### 6.2. Asteroid

Поля:

- id;
- position;
- velocity;
- radius/size;
- health;
- type;
- splitLevel;
- scoreValue.

Поведение:

- движение;
- столкновение с игроками;
- получение урона;
- разрушение;
- деление на меньшие астероиды;
- генерация бонусов с некоторой вероятностью.

### 6.3. Projectile

Поля:

- id;
- ownerPlayerId;
- position;
- velocity;
- damage;
- ttl;
- createdAt.

Поведение:

- движение;
- столкновение с астероидами;
- исчезновение после попадания;
- исчезновение по TTL.

### 6.4. Bonus / Power-up

В MVP можно сделать один-два простых бонуса:

- восстановление здоровья;
- временное усиление стрельбы;
- щит;
- ускорение;
- дополнительное количество очков.

### 6.5. GameSession / Room

Поля:

- roomId;
- players;
- asteroids;
- projectiles;
- bonuses;
- status;
- startedAt;
- scoreTable;
- worldSeed;
- tickNumber.

Поведение:

- создание комнаты;
- подключение игроков;
- отключение игроков;
- игровой тик;
- рассылка состояния клиентам;
- завершение или рестарт сессии.

## 7. Карта и мир

Мир должен восприниматься как **условно бесконечный** или как очень большая 2D-локация.

Для MVP допустимы варианты:

### Вариант A. Большая ограниченная карта

Например:

```text
10000 x 10000 условных единиц
```

Плюсы:

- проще реализовать;
- проще синхронизировать;
- проще отлаживать;
- лучше для MVP.

### Вариант B. Торроидальный мир

Игрок, вылетевший за правую границу, появляется слева. Аналогично по вертикали.

Плюсы:

- ближе к классической Asteroids-механике;
- мир ощущается бесшовным;
- не нужно делать настоящую бесконечность.

### Вариант C. Chunk-based world

Мир делится на чанки, астероиды генерируются по seed.

Плюсы:

- ближе к «условно бесконечной» локации;
- архитектурно интереснее.

Минусы:

- сложнее для первого MVP;
- требует аккуратной синхронизации;
- выше риск увязнуть до появления играбельной версии.

**Рекомендация для агента:** начать с варианта A или B. Chunk-based генерацию рассматривать только после первого рабочего онлайн-MVP.

## 8. Сетевой режим

### 8.1. Обязательный принцип

Сервер должен быть **авторитетным**.

Клиент не должен сам решать:

- попал ли снаряд;
- сколько очков начислить;
- умер ли игрок;
- разрушился ли астероид;
- какой итоговый рейтинг.

Клиент отправляет на сервер только намерения игрока:

```text
turnLeft=true
turnRight=false
thrust=true
shoot=false
```

Сервер:

- применяет input;
- считает физику;
- проверяет столкновения;
- начисляет очки;
- обновляет состояние мира;
- отправляет снапшоты клиентам.

### 8.2. Минимальная модель обмена

Клиент → сервер:

```json
{
  "type": "input",
  "seq": 123,
  "roomId": "abc",
  "playerId": "p1",
  "input": {
    "left": false,
    "right": true,
    "thrust": true,
    "shoot": false
  }
}
```

Сервер → клиент:

```json
{
  "type": "snapshot",
  "tick": 456,
  "players": [],
  "asteroids": [],
  "projectiles": [],
  "bonuses": [],
  "scoreboard": []
}
```

### 8.3. Сетевые цели MVP

Для первой версии достаточно:

- WebSocket-соединение;
- одна или несколько комнат;
- 2–8 игроков в комнате;
- 10–20 обновлений состояния в секунду;
- клиентская интерполяция для плавности;
- базовая обработка disconnect/reconnect;
- отсутствие сложной авторизации.

### 8.4. Что не надо делать в первом MVP

Не нужно в первой версии:

- полноценный matchmaking;
- глобальные аккаунты;
- постоянный глобальный рейтинг;
- античит промышленного уровня;
- синхронизация на сотни игроков;
- распределённый мир;
- сложная rollback-схема;
- ECS-фреймворк, если он не нужен;
- микросервисная архитектура.

## 9. Очки и рейтинг внутри сессии

Минимальные правила:

| Событие | Изменение очков |
|---|---:|
| Попадание по астероиду | +1 |
| Разрушение маленького астероида | +5 |
| Разрушение среднего астероида | +10 |
| Разрушение большого астероида | +20 |
| Подбор бонуса | +2 |
| Получение урона | -2 |
| Смерть | -10 |
| Friendly fire, если будет включён | -5 |

Рейтинг считается в рамках комнаты/сессии.

В MVP рейтинг можно не сохранять между сессиями.

## 10. PvE как первый режим

Первый режим — PvE.

Игроки сражаются не друг с другом, а с окружением:

- астероиды;
- волны астероидов;
- усиленные астероиды;
- возможно, простые враждебные дроны на следующих этапах.

PvP можно рассматривать позже как отдельный режим, но не смешивать его с первым MVP.

## 11. Визуальный стиль

На первом этапе визуальный стиль не критичен.

Достаточно:

- чёрный/тёмный фон;
- корабль в виде треугольника/стрелки;
- астероиды в виде многоугольников;
- снаряды в виде точек/линий;
- простой HUD;
- таблица очков.

Главное — не застрять на графике до появления сетевого геймплея.

## 12. Управление

Для веб-MVP:

| Действие | Клавиша |
|---|---|
| Поворот влево | A или ArrowLeft |
| Поворот вправо | D или ArrowRight |
| Ускорение | W или ArrowUp |
| Торможение / reverse thrust | S или ArrowDown |
| Выстрел | Space |
| Пауза/debug overlay | P или F3 |

Для мобильного варианта, если он появится позже:

- виртуальный джойстик;
- отдельная кнопка стрельбы;
- возможно, автострельба в ближайшую цель как упрощение.

## 13. Архитектурные варианты, которые должен рассмотреть агент

Агент должен явно сравнить варианты реализации и выбрать один для MVP.

### Вариант 1. Web: TypeScript + Canvas/PixiJS + Cloudflare Workers/Durable Objects

Потенциальные плюсы:

- браузерный запуск по ссылке;
- хороший учебный эффект;
- один язык на клиенте и сервере;
- serverless/edge-деплой;
- удобно подключить GitHub;
- не нужно поднимать отдельный VPS.

Потенциальные минусы:

- надо разобраться с ограничениями runtime;
- WebSocket/game-loop в serverless требует аккуратной реализации;
- может быть сложнее отлаживать, чем локальный Node.js.

### Вариант 2. Web: TypeScript + Node.js WebSocket server + любой хостинг/VPS

Потенциальные плюсы:

- проще понять серверную часть;
- проще локально отлаживать;
- стандартный WebSocket-сервер;
- проще держать игровой loop в памяти.

Потенциальные минусы:

- нужен сервер/хостинг;
- нужно думать о деплое;
- больше эксплуатационной рутины.

### Вариант 3. Godot

Потенциальные плюсы:

- игровой движок из коробки;
- сцены, физика, ассеты;
- можно экспортировать под web/mobile/desktop;
- хорош для учебной игры.

Потенциальные минусы:

- отдельный стек;
- сеть и деплой всё равно надо проектировать отдельно;
- может быть лишним для простой Canvas-игры.

### Вариант 4. Unity

Потенциальные плюсы:

- мощный движок;
- много готовых решений;
- мобильный экспорт.

Потенциальные минусы:

- тяжелее;
- выше порог входа;
- для учебного минимального проекта может быть избыточен.

## 14. Предварительная рекомендация

Базовая рекомендация для первого прототипа:

```text
TypeScript monorepo
  /client  — браузерный клиент на Canvas 2D или PixiJS
  /server  — WebSocket game server
  /shared  — общие типы сообщений, DTO, game constants
```

Первый этап лучше сделать на локальном Node.js WebSocket server, чтобы быстро доказать механику.

После этого агент должен оценить перенос backend-части на Cloudflare Workers/Durable Objects или оставить Node.js-сервер, если так проще и надёжнее для учебного проекта.

## 16. Этапы разработки

### Этап 0. Аналитика и выбор стека

Агент должен:

1. Уточнить возможные технологические варианты.
2. Сравнить веб, mobile, Godot, Unity.
3. Выбрать стек MVP.
4. Зафиксировать решение в `docs/decisions/ADR-001-platform-choice.md`.
5. Зафиксировать сетевую модель в `docs/decisions/ADR-002-network-model.md`.

Результат:

- принятое архитектурное решение;
- понятный план работ;
- созданный репозиторий/скелет проекта.

### Этап 1. Одиночная локальная механика

Сделать локальную версию без сети:

- корабль;
- управление;
- инерция;
- астероиды;
- стрельба;
- коллизии;
- очки;
- смерть/респавн.

Критерий готовности:

- в браузере можно играть одному;
- астероиды уничтожаются;
- очки считаются.

### Этап 2. Локальный сетевой режим

Сделать сервер и подключение нескольких клиентов локально:

- WebSocket-сервер;
- комната;
- несколько игроков;
- общий мир;
- серверный game-loop;
- снапшоты клиентам.

Критерий готовности:

- два окна браузера подключаются к одной комнате;
- каждый игрок видит другого;
- астероиды общие;
- очки общие;
- сервер является источником истины.

### Этап 3. Онлайн-деплой

Сделать доступ через интернет.

Варианты:

- Cloudflare Pages + Workers/Durable Objects;
- Cloudflare Pages + внешний Node.js backend;
- другой минимальный хостинг.

Критерий готовности:

- есть публичная ссылка;
- два игрока с разных устройств могут зайти в одну комнату;
- игра синхронизируется онлайн.

### Этап 4. Полировка MVP

Добавить:

- стартовый экран;
- создание/выбор комнаты;
- ник игрока;
- таблицу очков;
- базовые бонусы;
- обработку отключений;
- простое логирование;
- debug overlay.

### Этап 5. Развитие после MVP

Возможные направления:

- волны астероидов;
- разные типы оружия;
- разные корабли;
- кооперативные цели;
- PvP-режим;
- постоянный рейтинг;
- аккаунты;
- мобильное управление;
- визуальные эффекты;
- звук;
- редактор параметров сессии;
- боты;
- replay/debug режим.

## 17. Технические требования к MVP

### 17.1. Производительность

Целевые ориентиры:

- 2–8 игроков в комнате;
- 50–200 астероидов;
- 20 серверных тиков в секунду на MVP;
- 60 FPS отрисовка на клиенте;
- не передавать лишние данные по сети;
- не отправлять полный мир чаще, чем нужно.

### 17.2. Синхронизация

Минимальный подход:

- серверный tick;
- клиент отправляет input;
- сервер отправляет snapshot;
- клиент интерполирует позиции;
- для своего корабля можно добавить простую client-side prediction.

### 17.3. Безопасность

Для MVP достаточно:

- сервер валидирует входящие сообщения;
- клиент не может напрямую начислить себе очки;
- клиент не может создать астероид или бонус;
- клиент не может отправить произвольную позицию корабля как истину;
- сервер ограничивает частоту сообщений.

### 17.4. Наблюдаемость

Добавить debug-информацию:

- FPS;
- ping;
- server tick;
- количество игроков;
- количество астероидов;
- количество снарядов;
- размер последнего snapshot;
- состояние WebSocket.

## 18. Протокол сообщений

Агент должен вынести протокол в общий пакет `shared`.

Минимальные типы сообщений:

### Client → Server

```ts
type ClientMessage =
  | JoinRoomMessage
  | LeaveRoomMessage
  | PlayerInputMessage
  | PingMessage;
```

### Server → Client

```ts
type ServerMessage =
  | JoinedRoomMessage
  | GameSnapshotMessage
  | PlayerDiedMessage
  | ScoreboardMessage
  | ErrorMessage
  | PongMessage;
```

## 19. Правила для агента/Codex

Агент должен работать итеративно.

### 19.1. Не пытаться сделать всё сразу

Запрещено начинать с большой архитектуры, аккаунтов, красивого UI, сложного движка и постоянного рейтинга.

Сначала нужен маленький, но настоящий end-to-end slice:

```text
игрок → управление → сервер → общий мир → астероид → выстрел → очки → второй игрок видит результат
```

### 19.2. После каждого этапа фиксировать результат

После каждого этапа агент должен обновлять:

- `README.md`;
- `docs/architecture.md`;
- `docs/roadmap.md`;
- список выполненных задач;
- список технического долга.

### 19.3. Делать проверяемые шаги

Каждая задача должна иметь критерий проверки.

Плохая задача:

```text
Сделать сеть.
```

Хорошая задача:

```text
Добавить WebSocket-сервер, который принимает два подключения, создаёт комнату default, рассылает снапшот с позициями игроков 20 раз в секунду. Проверка: открыть две вкладки браузера, убедиться, что движение одного игрока видно во второй вкладке.
```

### 19.4. Сначала простая графика

Не тратить время на ассеты до появления сетевого MVP.

Вся графика может быть примитивной:

- треугольник;
- окружности;
- линии;
- текст.

### 19.5. Вести решения как ADR

Все важные решения фиксировать в `docs/decisions`.

Минимум:

- выбор платформы;
- выбор сетевой модели;
- выбор деплоя;
- выбор игрового цикла;
- выбор модели мира.

## 21. Definition of Done для первого MVP

MVP считается готовым, если:

- игру можно открыть по ссылке;
- два игрока могут подключиться к одной комнате через интернет;
- оба игрока видят общий мир;
- корабли управляются независимо;
- астероиды существуют на сервере;
- выстрелы уничтожают астероиды;
- очки начисляются на сервере;
- смерть или урон дают штраф;
- таблица очков видна игрокам;
- игру можно перезапустить;
- в README есть инструкция запуска локально и онлайн;
- в docs зафиксирована архитектура.

## 23. Важное ограничение

Главный риск проекта — преждевременное усложнение.

## 24. Формулировка задачи для Codex

Сформируй и реализуй учебный проект онлайн-игры в духе Asteroids.

Сначала изучи это ТЗ, затем:

1. Предложи технологический стек для MVP.
2. Сравни минимум два варианта реализации:
   - Web + TypeScript + WebSocket;
   - Web + Cloudflare Pages/Workers/Durable Objects или другой online-first вариант.
3. Объясни, почему выбранный стек подходит для учебного проекта.
4. Создай структуру проекта.
5. Реализуй первый вертикальный срез:
   - клиент;
   - сервер;
   - подключение игрока;
   - управление кораблём;
   - общий мир;
   - астероиды;
   - выстрел;
   - начисление очков;
   - видимость изменений вторым клиентом.
6. После каждого этапа обновляй документацию.
7. Не усложняй проект до появления работающего онлайн-MVP.

Итоговая цель первого цикла разработки:

```text
Открыть игру в двух браузерах или на двух устройствах,
подключиться к одной комнате,
полетать двумя кораблями,
пострелять по общим астероидам,
увидеть общий рейтинг сессии.
```
